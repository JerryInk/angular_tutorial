## Неделя 1. Архитектура Angular 22 и экосистема разработки без Zone.js
------------------------------
## ЛЕКЦИЯ 1: Эволюция Angular и переход к архитектуре Signals-First
Продолжительность: 2 академических часа (90 минут)

## 1. Глубокий исторический экскурс: От AngularJS до Angular 22 (35 минут)
История Angular — это история борьбы за производительность, предсказуемость состояния и снижение объема загружаемого кода. Фреймворк прошел три монументальные технологические эпохи.

## Эпоха 1: AngularJS (2010–2016) — Рождение и концепция $digest
В 2010 году Мишко Хевери создал AngularJS (версии 1.х). Это был прорыв: веб-разработка получила двухстороннее связывание данных (Two-way Data Binding), директивы и MVC-архитектуру на клиенте.
Однако архитектура имела критический изъян — цикл $digest и механизм «грязной проверки» (dirty checking). Фреймворк циклически обходил все зарегистрированные переменные ($watchers), чтобы узнать, изменилось ли их значение. На больших интерфейсах (более 2000 элементов) приложение начинало необратимо лагать. Попытка исправить это привела к осознанию: проект нужно переписать с нуля.

## Эпоха 2: Angular 2 и концепция Zone.js (2016–2023)
В 2016 году вышел Angular 2 (его стали называть просто Angular, отбросив JS). Это был совершенно новый фреймворк, написанный на TypeScript, ориентированный на компонентный подход и использующий однонаправленный поток данных.
Для автоматического обновления интерфейса Google внедрил библиотеку Zone.js.

* Как это работало: Zone.js буквально «хакала» (monkey-patched) все асинхронные API браузера: ```setTimeout```, ```fetch```, ```addEventListener```, ```Promises```.
* Проблема: Как только происходил любой клик или завершался HTTP-запрос, Zone.js сигнализировала Angular: «Что-то произошло, пересчитай всё дерево компонентов сверху вниз!».
* Чтобы оптимизировать это, разработчикам приходилось вручную переключать компоненты на стратегию ```ChangeDetectionStrategy.OnPush``` и работать со сложной библиотекой асинхронных потоков RxJS. Архитектура стала тяжелой, а порог входа — колоссальным.

## Эпоха 3: Ренессанс Angular — От Ivy к Signals (2023–2026)
Начиная с версии 14, команда Angular запустила масштабный процесс «Ренессанса»:

* Angular 14–15 (2022): Появление Standalone-компонентов, которые позволили отказаться от громоздких и запутанных модулей (```@NgModule```).
* Angular 17 (2023): Внедрение нового синтаксиса шаблонов (Control Flow: ```@if```, ```@for```) и компилятора на базе Vite + Esbuild, ускорившего сборку в 10 раз. Появление Signals в режиме Developer Preview.
* Angular 18–20 (2024–2025): Стабилизация Сигналов. Внедрение концепции Zoneless (работа без Zone.js).
* Angular 22 (Текущее время, 2026): Исторический рубеж. Старый механизм Zone.js полностью исключен из шаблона генерации новых приложений. Произошел финальный переход на архитектуру Signals-First, где реактивность стала точечной (гранулярной). Интерфейс обновляет ровно тот DOM-узел, который связан с изменившимся Сигналом, не затрагивая другие компоненты.

## 2. Базовые сущности фреймворка в Angular 22 (20 минут)
Современная архитектура Angular 22 полностью избавилась от концепции модулей. Приложение — это граф независимых сущностей:

   1. Standalone-компоненты: Самодостаточные UI-блоки. Они сами декларируют, какие сторонние компоненты или пайпы им нужны для рендеринга, через свойство ```imports: [...]``` прямо в декораторе ```@Component```.
   2. Директивы: Сущности без собственных шаблонов. Они внедряются в существующие HTML-теги для управления их поведением (например, директива для валидации или кастомного дропдауна).
   3. Пайпы (Pipes): Инструменты для декларативного преобразования данных на лету в коде шаблона (например, форматирование даты под локаль пользователя).
   4. Сервисы и функция ```inject()```: Классы для бизнес-логики. В Angular 22 повсеместно используется функциональный DI (Dependency Injection) через ```inject(MyService)``` вместо громоздких конструкторов.

## 3. Анатомия проекта и Expressive-шаблоны v22 (20 минут)
В Angular 22 шаблонизатор получил сильные нативные расширения, сближающие его с современным JavaScript:

* Поддержка Spread-оператора ```(...)```: Прямо внутри HTML-шаблона можно передавать деструктурированные объекты в инпуты компонентов.
* Стрелочные функции в шаблоне: Простые трансформации или предикаты можно описывать прямо в разметке, не засоряя TypeScript-файл компонента методами-пустышками.

## 4. Эволюция сборки: Полный отказ от Webpack (15 минут)
Долгое время Angular ассоциировался с тяжелой и медленной сборкой на Webpack. В Angular 22 поддержка Webpack полностью прекращена. Новый application-билдер использует связку:

* Esbuild: Для сверхбыстрой компиляции TypeScript кода и оптимизации продакшен-бандла (Tree Shaking, удаление мертвого кода).
* Vite: Для мгновенного запуска локального сервера разработки и горячей замены модулей (HMR) без перезагрузки страницы.

## ПРАКТИЧЕСКОЕ ЗАНЯТИЕ 1: Инициализация Zoneless-приложения Angular 22
Продолжительность: 2 академических часа (90 минут)
Цель работы: Развернуть современное рабочее окружение, создать чистый Zoneless-проект на базе Angular 22, изучить новую структуру конфигурации и создать первый Standalone-компонент с использованием синтаксиса v22.

## Шаг 1: Инспекция и подготовка Node.js окружения (15 минут)

   1. Убедиться, что на компьютерах установлена актуальная LTS-версия Node.js (v22.0.0 или выше). Команда в терминале:
```bash
  node -v
```
   2. Установить или обновить глобальный Angular CLI до актуальной 22-й версии:
```bash
  npm install -g @angular/cli@latest
```
   3. Проверить успешность установки: ```ng version```. Нужно зафиксировать, что в строке Angular CLI и Node отображаются версии 22+.

## Шаг 2: Создание проекта новой генерации (20 минут)

   1. Создать каркас приложения под выбранную тему (флаг ```--standalone``` больше не нужен, он включен по умолчанию; отключаем SSR для упрощения локального старта):
```bash
  ng new angular-22-project --style=scss --routing --ssr=false
```
   2. Перейти в папку проекта: ```cd angular-22-project```.
   3. Открыть файл src/app/app.config.ts. Обратить внимание на отсутствие конфигураций для zone.js и наличие современного провайдера реактивности без зон:
```typescript
  import { ApplicationConfig, provideZonelessChangeDetection } from '@angular/core';
  import { provideRouter } from '@angular/router';
  import { routes } from './app.routes';

  export const appConfig: ApplicationConfig = {
    providers: [
      provideZonelessChangeDetection(), // Активирует Signals-First архитектуру без Zone.js
      provideRouter(routes)
    ]
  };
```

## Шаг 3: Перевод корневого компонента на OnPush (15 минут)

   1. Открыть src/app/app.component.ts.
   2. В мире без зон стратегия ```ChangeDetectionStrategy.OnPush``` важна для точечного обновления экрана. Добавить её в метаданные:
```typescript   
  import { Component, ChangeDetectionStrategy } from '@angular/core';
  import { RouterOutlet } from '@angular/router';

  @Component({
    selector: 'app-root',
    imports: [RouterOutlet],
    templateUrl: './app.component.html',
    styleUrl: './app.component.scss',
    changeDetection: ChangeDetectionStrategy.OnPush // Фиксируем OnPush архитектуру
  })
  export class AppComponent {}
```
   3. Открыть src/app/app.component.html, удалить сгенерированную разметку Google, оставив только:
```html
  <main class="app-container">
    <router-outlet />
  </main>
```

## 💡 Зачем нужен OnPush в мире Angular 22 (Zoneless)?
При переходе на Zoneless-архитектуру (без Zone.js) поведение Angular принципиально меняется, и явное указание ```ChangeDetectionStrategy.OnPush``` решает критически важную задачу.

## Как было раньше (с Zone.js):
По умолчанию использовалась стратегия Default. Библиотека Zone.js перехватывала абсолютно любое событие в браузере (клик, таймер, сетевой запрос) и заставляла Angular сканировать абсолютно все компоненты сверху вниз, проверяя, не изменилось ли там что-то. Это работало автоматически, но создавало колоссальную нагрузку на процессор.

## Что происходит в Angular 22 (Zoneless + OnPush):
Когда мы убираем Zone.js, Angular больше не может «шпионить» за браузером. Теперь фреймворку нужно точно знать, какой именно компонент заслуживает перерисовки.
Стратегия ```ChangeDetectionStrategy.OnPush``` сообщает движку Angular:

«Не проверяй этот компонент вхолостую. Обновляй его шаблон исключительно тогда, когда изменятся его входные параметры (inputs), произойдет его собственное событие (например, клик внутри него) или внутри него явно изменится Сигнал (Signal)».

## Почему это критично:

   1. Точечный рендеринг (Гранулярность): Без ```OnPush``` в Zoneless-режиме Angular все еще может пытаться выполнять лишние проверки верхнеуровневых узлов при некоторых глобальных событиях. ```OnPush``` жестко изолирует компонент.
   2. Оптимизация по умолчанию: Приучаясь писать ```changeDetection: ChangeDetectionStrategy.OnPush```, мы закладываем фундамент для создания молниеносных enterprise-приложений. Шаблон компонента будет обновляться микро-дельтами — только в тех местах, где поменялся конкретный Сигнал.
   3. Совместимость с Signals-First: Вся новая экосистема Angular 22 (включая Signal Forms и Resource API) спроектирована с расчетом на то, что компоненты работают в режиме ```OnPush```.

## Шаг 4: Создание Standalone-компонента и применение фич v22 (25 минут)

   1. Сгенерировать компонент приветственной панели с помощью CLI:
```bash
  ng g c components/welcome-panel
```
   2. Открыть welcome-panel.component.ts и добавить базовые свойства для проверки работы нового шаблонизатора:
```typescript
  import { Component } from '@angular/core';
   
  @Component({
    selector: 'app-welcome-panel',
    templateUrl: './welcome-panel.component.html',
    styleUrl: './welcome-panel.component.scss'
  })
   export class WelcomePanelComponent {
    // Имитация состояния бэкенда для проверки множественного @case
    appState = 'maintenance'; 
    studentConfig = { name: 'Иван', group: 'ИТ-301' };
  }
```
   3. Открыть welcome-panel.component.html и написать разметку с использованием сгруппированного ```@switch``` (фича Angular 22):
```html
  <div class="panel-card">
    <h2>Система электронного обучения (Angular 22)</h2>
    <p>Студент: {{ studentConfig.name }} (Группа: {{ studentConfig.group }})</p>
   
    <!-- Демонстрация синтаксиса Angular 22 с группировкой кейсов -->
    @switch (appState) {
      @case ('loading')
      @case ('maintenance') {
        <div class="status-alert processing">
          <p>Внимание: Базовый бэкенд находится на обслуживании или загружается.</p>
        </div>
      }
      @case ('ready') {
        <div class="status-alert success">
          <p>Соединение с REST API успешно установлено!</p>
        </div>
      }
      @default {
        <p>Статус подсистемы не определен.</p>
      }
    }
  </div>
```
   
## Шаг 5: Монтирование и запуск (15 минут)

   1. Импортировать WelcomePanelComponent в массив ```imports``` файла src/app/app.component.ts.
   2. Вставить селектор ```<app-welcome-panel />``` в app.component.html.
   3. Запустить проект через ```ng serve```.
   4. Открыть вкладку Network в инструментах разработчика браузера и показать студентам, что среди загружаемых скриптов больше нет файла zone.js, а само приложение работает стабильно и мгновенно реагирует на изменения кода.
