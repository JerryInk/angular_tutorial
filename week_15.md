## Неделя 15. Модульное тестирование (Unit Testing)
------------------------------
## ЛЕКЦИЯ 15: Модульное тестирование Angular-приложений в Zoneless/OnPush архитектуре
Продолжительность: 2 академических часа (90 минут)

## 1. Философия тестирования: Зачем ИТ-инженеру Unit-тесты? (20 минут)
Модульное (Unit) тестирование — это процесс проверки отдельных, изолированных минимальных единиц кода (функций, пайпов, сервисов, компонентов) в полной изоляции от внешнего мира.
Инженерная ценность тестов:

* Регрессионная безопасность: При изменении или оптимизации кода (рефакторинге) запуск тестов мгновенно покажет, не сломалась ли старая бизнес-логика.
* Документирование кода: Хорошо написанный Unit-тест наглядно демонстрирует другому разработчику, как должен вести себя модуль при различных входных данных.
* Экосистема в Angular 22: Современный Angular поставляется с готовой инфраструктурой тестирования. Тестовые сценарии описываются в файлах с расширением .spec.ts, которые CLI автоматически создает рядом с каждым компонентом.

## 2. Инструментарий тестирования: Jasmine, Karma и утилита TestBed (25 минут)

* Jasmine: Популярный Behavior-Driven Development (BDD) фреймворк для написания тестов. Он предоставляет синтаксические конструкции для описания тест-кейсов (describe, it) и проверки утверждений (expect).
* TestBed: Главный служебный инструмент Angular для тестирования. Он создает искусственный, контролируемый Angular-модуль (динамический инжектор), в котором эмулируется жизненный цикл тестируемого компонента или сервиса.

Базовая структура тест-кейса Jasmine:
```typescript
describe('Тестирование модуля X', () => {
  it('должен возвращать корректную строку при условии Y', () => {
    const result = myFunction('Y');
    expect(result).toBe('Ожидаемый ответ'); // Проверка утверждения
  });
});
```

## 3. Изоляционное тестирование утилит (Пайпов и Сервисов) (20 минут)
Тестирование чистых функций, пайпов и простых сервисов называется изоляционным. Для его проведения нет необходимости разворачивать тяжелый контекст TestBed и эмулировать HTML-шаблоны. Пайп или сервис можно протестировать как обычный JavaScript-класс, создав его экземпляр через ключевое слово ```new```, что выполняется мгновенно и экономит ресурсы процессора.

## 4. Специфика тестирования OnPush-компонентов в Zoneless-режиме (25 минут)
В Angular 22, когда приложение работает без Zone.js, традиционный метод ```fixture.detectChanges()``` в тестах компонентов ведет себя иначе.

* Поскольку OnPush-компоненты завязаны на Сигналы, Angular не будет автоматически перерисовывать HTML-шаблон теста при простом изменении полей класса.
* Новый стандарт v22: Чтобы зафиксировать изменения Сигнала в тестовом шаблоне, необходимо использовать встроенную утилиту ```fixture.componentRef.setInput(...)``` для входных данных или явно вызывать метод ```.set()``` у тестируемого Сигнала, после чего вызывать метод синхронизации UI — ```await fixture.whenStable()```.

## ПРАКТИЧЕСКОЕ ЗАНЯТИЕ 15: Написание Unit-тестов для кастомного пайпа и OnPush-компонента
Продолжительность: 2 академических часа (90 минут)
Цель работы: Освоить запуск тестовой среды через Angular CLI, написать изоляционный Unit-тест для кастомного пайпа ```TruncateTextPipe``` и разработать интеграционный тест для OnPush-компонента с проверкой рендеринга Сигналов.

## Шаг 1: Знакомство с тестовой средой и запуск CLI (15 минут)

   1. Студенты открывают терминал проекта и запускают тестовый раннер Angular:
```bash
  ng test
```
   2. Angular CLI скомпилирует приложение и откроет окно браузера (Karma), где в реальном времени отобразится отчет обо всех текущих тестах проекта. Раннер переходит в режим ожидания (watch): при любом изменении файлов .spec.ts тесты будут перезапускаться автоматически.

## Шаг 2: Написание изоляционного теста для Пайпа (25 минут)
Откроем файл src/app/pipes/truncate-text.pipe.spec.ts и напишем строгие проверки для различных граничных условий обрезки строк.
```typescript
  import { TruncateTextPipe } from './truncate-text.pipe';
  
  describe('TruncateTextPipe (Изоляционное тестирование)', () => {
    let pipe: TruncateTextPipe;
  
    // Данный блок выполняется один раз перед каждым тест-кейсом 'it'
    beforeEach(() => {
      pipe = new TruncateTextPipe(); // Создаем чистый экземпляр класса напрямую
    });
  
    it('должен успешно создавать экземпляр пайпа', () => {
      expect(pipe).toBeTruthy();
    });
  
    it('должен возвращать пустую строку, если на вход пришел null или undefined', () => {
      expect(pipe.transform('')).toBe('');
    });
  
    it('не должен обрезать строку, если её длина меньше указанного лимита', () => {
      const shortText = 'Короткий текст';
      // Проверяем, что текст остался неизменным при лимите в 20 символов
      expect(pipe.transform(shortText, 20)).toBe(shortText); 
    });
  
    it('должен корректно обрезать длинную строку и добавлять троеточие в конец', () => {
      const longText = 'Длинное инженерное описание лабораторного курса';
      const result = pipe.transform(longText, 10);
      
      // Ожидаем, что строка обрежется до 10 символов + '...'
      expect(result).toBe('Длинное ин...');
    });
  });
```

## Шаг 3: Настройка интеграционного теста OnPush-компонента (25 минут)
Напишем тест для простого компонента, отображающего статус счетчика на Сигналах.

Сгенерируем тестовый виджет: ```ng g c components/test-counter --standalone```.
Открыть test-counter.component.ts:
```typescript
  import { Component, ChangeDetectionStrategy, signal } from '@angular/core';
  
  @Component({
    selector: 'app-test-counter',
    standalone: true,
    template: `
      <div class="counter-box">
        <span class="count-value">{{ count() }}</span>
        <button (click)="increment()">Плюс</button>
      </div>
    `,
    changeDetection: ChangeDetectionStrategy.OnPush
  })
  export class TestCounterComponent {
    count = signal<number>(0);
    increment() { this.count.update(c => c + 1); }
  }
```

## Шаг 4: Реализация тестовых сценариев для Сигналов (15 минут)
Открыть автоматически созданный файл test-counter.component.spec.ts и переписать его структуру под проверку Zoneless/OnPush поведения:
```typescript
  import { ComponentFixture, TestBed } from '@angular/core/testing';
  import { TestCounterComponent } from './test-counter.component';
  import { By } from '@angular/platform-browser';
  
  describe('TestCounterComponent (Тестирование OnPush и Сигналов)', () => {
    let component: TestCounterComponent;
    let fixture: ComponentFixture<TestCounterComponent>;
  
    beforeEach(async () => {
      // Настраиваем тестовое окружение Angular TestBed
      await TestBed.configureTestingModule({
        imports: [TestCounterComponent] // Подключаем наш Standalone-компонент
      }).compileComponents();
  
      fixture = TestBed.createComponent(TestCounterComponent);
      component = fixture.componentInstance; // Ссылка на TS-класс компонента
      fixture.detectChanges(); // Первичный рендеринг HTML
    });
  
    it('должен отображать стартовое значение Сигнала (0) в HTML разметке', () => {
      // Находим DOM-элемент с помощью css-селектора класса
      const countElement = fixture.debugElement.query(By.css('.count-value')).nativeElement as HTMLElement;
      expect(countElement.textContent).toContain('0');
    });
  
    it('должен корректно обновлять HTML при изменении Сигнала и вызове клика', async () => {
      const buttonElement = fixture.debugElement.query(By.css('button')).nativeElement as HTMLButtonElement;
      
      // Имитируем клик пользователя по кнопке
      buttonElement.click();
      
      // Так как проект Zoneless, принудительно синхронизируем граф Сигналов с DOM шаблона
      fixture.detectChanges();
      await fixture.whenStable();
  
      const countElement = fixture.debugElement.query(By.css('.count-value')).nativeElement as HTMLElement;
      
      // Проверяем, что значение в HTML стало равным 1
      expect(component.count()).toBe(1); // Проверка стейта в классе
      expect(countElement.textContent).toContain('1'); // Проверка отображения на экране
    });
  });
```

## Шаг 5: Анализ результатов выполнения тестов (10 минут)

   1. Проверяем окно браузера Karma. Все написанные тесты (как для пайпа, так и для компонента счетчика) должны загореться зеленым цветом (SUCCESS).
   2. Если специально изменить ожидаемое значение в ```expect(result).toBe('Неправильный текст')```, тест мгновенно упадет, подсветив строку ошибки, что наглядно доказывает работоспособность системы контроля качества кода.
