## ЛЕКЦИЯ 4: Глубокое погружение в Angular Signals — Реактивный фундамент v22
Продолжительность: 2 академических часа (90 минут)

## 1. Анатомия Сигнала: Writable Signals (25 минут)
Angular Signal — это реактивная обертка вокруг значения, которая умеет уведомлять заинтересованных потребителей (шаблоны, другие сигналы) о своем изменении. В отличие от старых подходов (например, RxJS BehaviorSubject), сигналы не требуют явной подписки и отписки — Angular автоматически отслеживает, где именно был вызван сигнал.
Существует два типа сигналов: Writable (доступные для записи) и Read-Only (доступные только для чтения).
Создание Writable Signal:
```typescript
  count = signal<number>(0); // Начальное значение — 0
```
Для изменения значения Writable Signal используются два ключевых метода:

* ``` set(newValue) ``` — полностью перезаписывает значение новым:
```typescript
  this.count.set(10);
```
* ``` update(fn) ``` — обновляет значение на основе его текущего состояния (принимает чистую функцию):
```typescript
  this.count.update(current => current + 1);
```

Чтобы прочитать значение сигнала (как в TypeScript, так и в HTML-шаблоне), его необходимо вызвать как функцию: ```this.count()```.

## 2. Вычисляемые сигналы: Декларативный computed() (20 минут)
Если значение зависит от одного или нескольких других сигналов, используется функция ```computed()```. Она создает Read-Only Signal:
```typescript
  doubleCount = computed(() => this.count() * 2);
```
Критически важные свойства ```computed()```:

* Мемоизация (Кэширование): Значение внутри computed вычисляется только тогда, когда меняется исходный сигнал count. Если вы вызовете ```doubleCount()``` в шаблоне 10 раз, тяжелые вычисления внутри выполнятся всего один раз.
* Динамическое отслеживание зависимостей: Если внутри ```computed``` есть условие ```if```, Angular будет отслеживать зависимости только в активной ветке кода. Если ветка меняется, ненужные сигналы исключаются из графа зависимостей автоматически.

## 3. Побочные эффекты через effect() (20 минут)
Функция ```effect()``` создает операцию, которая автоматически перезапускается каждый раз, когда изменяются сигналы, прочитанные внутри нее.
```typescript
  constructor() {
    effect(() => {
      console.log(`Текущий счетчик равен: ${this.count()}`);
    });
  }
```
Правила использования ```effect()``` в Angular 22:

   1. Только в контексте инъекции: Эффекты должны объявляться там, где доступен InjectionContext (обычно это конструктор класса или инициализация полей). Если нужно запустить эффект динамически, придется вручную передавать Injector.
   2. Запрет на запись по умолчанию: Попытка вызвать ```signal.set()``` или ```signal.update()``` внутри ```effect()``` вызовет ошибку. Это сделано для предотвращения бесконечных циклов перерисовок. Для обхода (в редких архитектурных случаях) требуется флаг ```allowSignalWrites: true``` в конфигурации эффекта.
   3. Сферы применения: Логирование, синхронизация данных с ```localStorage```, кастомная аналитика или работа со сторонними не-Angular библиотеками (например, отрисовка графиков на Canvas/Chart.js при изменении данных).

## 4. Сигналы против классического Change Detection (25 минут)

| Критерий | Старый подход (Zone.js + Default) | Новый подход (Zoneless + Signals + OnPush) |
|---|---|---|
| Зона проверки | Проверяется всё дерево компонентов сверху вниз. | Обновляется только конкретный DOM-узел, связанный с Сигналом. |
| Затраты CPU | Высокие. Лишние вычисления при любом движении мыши. | Минимальные. Строго точечные ленивые обновления. |
| Порог входа | Низкий на старте, экстремально сложный при оптимизации. | Средний на старте, архитектура сразу оптимальна. |

## ПРАКТИЧЕСКОЕ ЗАНЯТИЕ 4: Интерактивный калькулятор бюджета проекта / Корзина задач
Продолжительность: 2 академических часа (90 минут)
Цель работы: На практике освоить работу с Writable Signals (set, update), реализовать каскад зависимостей через ```computed``` и настроить автоматическое сохранение состояния в ```localStorage``` через ```effect``` в режиме OnPush.

## Шаг 1: Инициализация компонента и модели данных (15 минут)

   1. Сгенерировать компонент менеджера бюджета проекта (или корзины):
```bash
  ng g c components/budget-calc
```
   2. Открыть budget-calc.component.ts и перевести его в OnPush:
```typescript
  import { Component, ChangeDetectionStrategy, signal, computed, effect } from '@angular/core';
  interface BudgetItem {
    id: number;
    name: string;
    cost: number;
  }
   
  @Component({
    selector: 'app-budget-calc',
    imports: [],
    templateUrl: './budget-calc.component.html',
    styleUrl: './budget-calc.component.scss',
    changeDetection: ChangeDetectionStrategy.OnPush
  })
  export class BudgetCalcComponent {
    // Инициализируем Writable Signal для списка расходов
    items = signal<BudgetItem[]>([
      { id: 1, name: 'Лицензии на ПО (Angular UI)', cost: 15000 },
      { id: 2, name: 'Хостинг базового бэкенда', cost: 5000 }
    ]);
  
    // Сигнал для ввода новой позиции
    newName = signal<string>('');
    newCost = signal<number>(0);
  }
``` 
   
## Шаг 2: Реализация зависимостей через ```computed()``` (20 минут)
Добавим реактивные вычисления, которые будут автоматически пересчитывать общую стоимость проекта и сумму налогов/надбавок, не требуя ручного вызова методов пересчета.
```typescript
  // Внутри класса BudgetCalcComponent

  // Вычисляемый сигнал: общая сумма (Read-Only)
  totalCost = computed(() => {
    console.log('--- Перерасчет totalCost (мемоизация в действии) ---');
    return this.items().reduce((sum, item) => sum + item.cost, 0);
  });

  // Вычисляемый сигнал: налог (20% от общей суммы)
  taxAmount = computed(() => this.totalCost() * 0.20);

  // Вычисляемый сигнал: итоговый бюджет
  grandTotal = computed(() => this.totalCost() + this.taxAmount());
```

## Шаг 3: Настройка синхронизации через effect() (15 минут)
Добавим автоматическое сохранение списка расходов в ```localStorage``` браузера при каждом изменении сигнала ```items```. Напишем эффект в конструкторе.
```typescript
  constructor() {
    effect(() => {
      // Этот код сработает автоматически при любом изменении этого.items() (добавление/удаление)
      const dataString = JSON.stringify(this.items());
      localStorage.setItem('project_budget', dataString);
      console.log('Данные успешно синхронизированы с localStorage!');
    });
  }
```
Дополнительно: в ```ngOnInit``` можно добавить чтение из ```localStorage``` для восстановления состояния при перезагрузке.

## Шаг 4: Бизнес-логика обновления Сигналов (15 минут)
Реализуем методы добавления и удаления элементов, используя синтаксис мутации сигналов.
```typescript
  addItem(): void {
    if (!this.newName().trim() || this.newCost() <= 0) return;

    const newItem: BudgetItem = {
      id: Date.now(),
      name: this.newName(),
      cost: this.newCost()
    };

    // Обновляем сигнал через .update() — иммутабельное добавление
    this.items.update(currentItems => [...currentItems, newItem]);

    // Сбрасываем поля ввода
    this.newName.set('');
    this.newCost.set(0);
  }

  removeItem(id: number): void {
    // Удаляем элемент через .update()
    this.items.update(currentItems => currentItems.filter(item => item.id !== id));
  }

  // Методы для двухстороннего связывания полей ввода вручную (до изучения Signal Forms)
  updateName(event: Event): void {
    const input = event.target as HTMLInputElement;
    this.newName.set(input.value);
  }

  updateCost(event: Event): void {
    const input = event.target as HTMLInputElement;
    this.newCost.set(Number(input.value));
  }
```
## Шаг 5: Проектирование шаблона на Сигналах (25 минут)

   1. Открыть budget-calc.component.html:
```html
  <div class="budget-box">
    <h3>Калькулятор бюджета семестрового проекта</h3>

    <!-- Форма добавления (управляется сигналами) -->
    <div class="add-form">
      <input type="text" [value]="newName()" (input)="updateName($event)" placeholder="Название расхода">
      <input type="number" [value]="newCost()" (input)="updateCost($event)" placeholder="Стоимость (₽)">
      <button (click)="addItem()">Добавить статью</button>
    </div>

    <!-- Список элементов -->
    <ul class="items-list">
      @for (item of items(); track item.id) {
        <li>
          <span>{{ item.name }} — {{ item.cost }} ₽</span>
          <button (click)="removeItem(item.id)" class="btn-delete">❌</button>
        </li>
      } @empty {
        <p>Расходы отсутствуют.</p>
      }
    </ul>

    <!-- Финальные метрики (вычисляемые сигналы) -->
    <div class="summary-panel">
      <p>Чистая стоимость: <strong>{{ totalCost() }} ₽</strong></p>
      <p>НДС (20%): <strong>{{ taxAmount() }} ₽</strong></p>
      <hr>
      <h4>Итого к оплате: {{ grandTotal() }} ₽</h4>
    </div>
  </div>
``` 
   2. Подключить ```<app-budget-calc />``` в app.component.ts.
   3. Протестировать в браузере. Обратить внимание на консоль: строка ```--- Перерасчет totalCost ---``` вызывается строго при добавлении или удалении элементов, подтверждая умное кэширование Angular. Убедиться, что при перезагрузке вкладки Application -> LocalStorage данные обновляются на лету.

## 💡 Архитектурный паттерн Smart & Dumb Components (Container & Presentational)
При проектировании интерфейсов на Angular 22 разделение компонентов на «Умные» (Smart) и «Глупые» (Dumb / Презентационные) является индустриальным стандартом. Это позволяет избежать превращения кода в хаотичную массу и делает компоненты переиспользуемыми.

## 1. Глупые (Dumb / Презентационные) компоненты
Это изолированные UI-элементы, задача которых — просто красиво отобразить то, что им передали, и сообщить о действиях пользователя.

* Зависимости: Полностью изолированы. Они ничего не знают о бэкенде, REST API, сервисах или глобальном состоянии приложения.
* Связь с миром: Получают данные исключительно сверху от родителя через входные сигналы (```input()```) и отправляют события наружу через ```output()```.
* Преимущества: Их легко тестировать с помощью Unit-тестов, они идеально подходят для повторного использования в разных частях проекта (например, кастомная кнопка, карточка курса, модальное окно).
* Change Detection: Идеально работают в режиме ```ChangeDetectionStrategy.OnPush```, так как зависят только от чистых входных сигналов.

## 2. Умные (Smart / Контейнеры) компоненты
Это «мозг» конкретного экрана или страницы приложения. Они координируют работу глупых компонентов.

* Зависимости: Напрямую взаимодействуют с внешним миром. В них внедряются Angular-сервисы (```inject(ApiService)```), они отправляют HTTP-запросы к базовому бэкенду.
* Связь с миром: Они слушают события (```output()```) от дочерних глупых компонентов, обрабатывают их, обновляют свои локальные Сигналы (```signal()```) или отправляют измененные данные на сервер. Затем обновленные данные они спускают вниз в виде ```input()``.
* Особенности: Обычно содержат внутри себя разметку, состоящую преимущественно из тегов других (глупых) компонентов. Сами по себе они редко содержат сложную HTML-верстку элементов интерфейса.

## Сравнительная матрица:
| Критерий | Smart Component (Контейнер) | Dumb Component (Презентационный) |
|---|---|---|
| Главная цель | Как приложение работает (бизнес-логика). | Как приложение выглядит (UI/отображение). |
| Доступ к сервисам/API | Да (через ```inject()```). | Нет (строгий запрет). |
| Как получает данные | Загружает с бэкенда или из общего хранилища. | Из родительского компонента через ```input()```. |
| Как передает действия | Вызывает методы сервисов, меняет состояние. | Излучает событие наружу через ```output()```. |
| Переиспользуемость | Низкая (привязан к конкретному экрану). | Высокая (можно переносить между проектами). |

## Реальный пример из практики (Тема: Список задач)
* Smart: TaskListComponent — запрашивает массив задач с сервера, хранит его в сигнале. При клике на удаление отправляет DELETE-запрос на бэкенд.
* Dumb: TaskCardComponent — принимает одну конкретную задачу через ```input()```. Рендерит её текст и дедлайн. При клике на крестик просто вызывает ```this.deleteEvent.emit()```, не зная, как именно задача удалится из базы данных.
