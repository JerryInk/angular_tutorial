## ЛЕКЦИЯ 16: Компиляция Ahead-of-Time (AOT), оптимизация бандла и деплой SPA
Продолжительность: 2 академических часа (90 минут)

## 1. Процесс компиляции Ahead-of-Time (AOT) (20 минут)
В Angular 22 компиляция по умолчанию происходит в режиме Ahead-of-Time (AOT) на этапе сборки проекта.

* Как это работает: Специальный компилятор Angular (@angular/compiler-cli) анализирует TypeScript-классы и сопоставляет их с HTML-шаблонами. Он превращает декларативную разметку (включая управляющие конструкции ```@if``` и ```@for```) в высокооптимизированный, чистый императивный JavaScript-код.
* Преимущества: Браузер пользователя не тратит ресурсы процессора на парсинг шаблонов «на лету». Ошибки в верстке или типах данных шаблона выявляются на этапе компиляции, а не в рантайме у пользователя.

## 2. Оптимизация бандла: Tree Shaking и Dead Code Elimination (20 минут)
Благодаря переходу на современный application-билдер (работающий на базе Esbuild), Angular 22 демонстрирует рекордные показатели веса продакшен-бандла:

* Tree Shaking («Встряхивание дерева»): Процесс статического анализа графа зависимостей. Если в проекте импортирован тяжелый сервис или функция, но они физически ни разу не вызываются в коде, сборщик полностью исключит их из итоговых .js файлов.
* Минификация и Обфускация: Код сжимается, удаляются пробелы, комментарии, а имена переменных и функций сокращаются до одной буквы для минимизации веса файлов, передаваемых по сети.

## 3. Архитектурная проблема маршрутизации SPA при перезагрузке (25 минут)
Необходимо разобрать важнейший нюанс развертывания SPA на классических веб-серверах (Nginx, Apache):

* Если пользователь кликает по ссылкам внутри Angular, роутер перехватывает пути (например, '/dashboard' или '/account'), динамически подменяя компоненты.
* Но если пользователь нажмет кнопку «Перезагрузить страницу» (F5) на URL ''/account', браузер минует Angular и отправит прямой запрос на физический сервер: «Дай мне файл по пути '/account'».
* Так как Angular — это Single Page Application, на сервере физически существует только один файл — index.html. Сервер не найдет папку '/account' и вернет ошибку 404 Not Found.
* Решение: Настройка Fallback (перенаправления) на стороне веб-сервера. Сервер должен быть сконфигурирован так, чтобы на абсолютно любой входящий запрос (клиентский URL) он всегда возвращал корневой файл index.html. Получив этот файл, Angular Router сам считает текущий URL из адресной строки и отобразит нужный ленивый компонент.

## 4. Критерии оценки и структура защиты финального проекта (25 минут)
Каждый студент должен продемонстрировать:

   1. Архитектурную чистоту кода (отсутствие Zone.js, использование ```ChangeDetectionStrategy.OnPush```, паттерн Smart/Dumb).
   2. Работу с реактивным состоянием (глобальный или локальные Signals Store, использование computed).
   3. Корректную валидацию реактивных форм и обработку ошибок.
   4. Успешную асинхронную интеграцию с выданной базовой частью REST API бэкенда.

## ПРАКТИЧЕСКОЕ ЗАНЯТИЕ 16: Продакшен-сборка приложения и демонстрация проектов
Продолжительность: 2 академических часа (90 минут)
Цель работы: Выполнить оптимизированную компиляцию проекта для продакшена, проанализировать состав и размеры получившихся JavaScript-чанков, защитить разработанное SPA-приложение перед преподавателем.

## Шаг 1: Запуск Production-сборки (15 минут)

   1. Останавливаем локальный dev-сервер разработки и запускают команду компиляции:
```bash
  ng build
```

   1. Современный билдер Angular 22 начнет параллельную оптимизацию кода. На экране отобразится отчет со списком сгенерированных чанков, их размерами и хэш-ключами (для предотвращения кэширования старого кода в браузерах клиентов).

## Шаг 2: Анализ состава бандла (15 минут)
Открываем появившуюся в корне проекта папку dist/[имя-проекта]/browser. Изучаем структуру дистрибутива:

* index.html — единственный физический HTML-файл, точка входа.
* main-[hash].js — основной код приложения (ядро Angular 22, DI, базовые сервисы). Обратите внимание на его минимальный вес благодаря отсутствию полифилов зон.
* styles-[hash].css — скомпилированные и минифицированные стили всего приложения.
* Набор файлов вида chunk-[hash].js — это изолированные ленивые страницы (Lazy Components). Они будут скачиваться браузером клиента строго по требованию.

## Шаг 3: Демонстрация и защита финальных работ (50 минут)
## Чек-лист проверки кода:

* Конфигурация проекта: В app.config.ts присутствует ```provideZonelessChangeDetection()```. Файлы zone.js полностью отсутствуют в зависимостях.
* Компоненты: Все UI-компоненты имеют явную декларацию ```changeDetection: ChangeDetectionStrategy.OnPush```.
* Реактивность: Отсутствуют прямые мутации массивов. Изменения состояния происходят через ```.set()``` или ```.update()``` у Signals. Поля форм или вычисляемые метрики завязаны на ```computed()```.
* Сетевой слой: Все запросы к бэкенду обернуты в Angular-сервисы с использованием функционального ```inject(HttpClient)```. Реализован функциональный интерцептор авторизации.
* Маршрутизация: Маршруты страниц загружаются лениво через ```loadComponent```. Параметры URL считываются современным способом через сигнальные инпуты (```input()```).
* Качество кода: Наличие зеленых Unit-тестов (```ng test```) для кастомных пайпов или сервисов.
